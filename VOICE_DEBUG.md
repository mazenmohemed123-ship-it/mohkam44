## 1. الإعدادات الحالية لـ `recognition`

تم تعيين الإعدادات كالتالي في الدالة `startListen` (الأسطر 60-73):

* **`continuous`**: قيمته الحالية هي `true` (`recognition.continuous = true;`).
  * *ماذا تعني:* نظرياً، تعني أن ميزة التعرف على الصوت يجب أن تستمر في الاستماع حتى لو صمت المستخدم أو توقف مؤقتاً، ولا تتوقف بعد أول جملة قصيرة.
* **`interimResults`**: قيمته الحالية هي `true` (`recognition.interimResults = true;`).
  * *ماذا تعني:* تعني أن المتصفح يرسل النتائج المؤقتة (الكلمات غير المكتملة أو قيد المعالجة) ويقوم بتحديثها بشكل لحظي في الواجهة قبل أن يستقر على النص النهائي.
* **`lang`**: يتم تعيينها ديناميكياً بناءً على اللغة المختارة من القائمة عبر الدالة `getLangCode(voiceLang)` (`recognition.lang = getLangCode(voiceLang);`).
  * *تفصيل:* القيمة الافتراضية لـ `voiceLang` هي `'ar'` (عربي)، وبالرجوع لجدول اللغات:
    ```typescript
    ar: 'ar-EG',      // عربي مصري
    ```
    يتم تمرير القيمة `'ar-EG'` إلى الـ API للتعرف على اللهجة المصرية.

## 2. ما الذي يحدث في الأحداث `onresult` و `onend` و `onerror`؟

### أ. حدث النتيجة `onresult`
```typescript
recognition.onresult = (event: any) => {
  const transcript = Array.from(event.results)
    .map((result: any) => result[0].transcript)
    .join('');
  setTranscript(transcript);
};
```
* **الوظيفة:** عندما يلتقط الميكروفون كلاماً، يتم تحويل النتائج القادمة من المتصفح `event.results` إلى مصفوفة، ثم استخراج النص من كل نتيجة (`result[0].transcript`)، ثم دمج النصوص معاً بسلسلة واحدة وتحديث الحالة `transcript` لعرضها فوراً في الواجهة.

### ب. حدث انتهاء التسجيل `onend`
```typescript
recognition.onend = () => {
  setMode(prev => prev === 'listening' ? 'idle' : prev);
};
```
* **الوظيفة:** عندما تنتهي خدمة التعرف على الصوت لأي سبب (مثل توقف المتصفح عن الاستماع، أو انتهاء الجلسة)، يتم فحص حالة المود (`mode`). إذا كانت الحالة الحالية هي الاستماع `'listening'`، يتم تغييرها فوراً إلى الوضع الخامل `'idle'` (مما يغير شكل الميكروفون والـ UI كأن التسجيل توقف).

### ج. حدث الخطأ `onerror`
```typescript
recognition.onerror = () => setMode('idle');
```
* **الوظيفة:** عند حدوث أي خطأ (مثل رفض صلاحيات الميكروفون، عدم وجود اتصال بالإنترنت للترجمة، أو مشاكل في المتصفح)، يتم تحويل الحالة فوراً إلى خامل `'idle'`.

## 3. هل يتم استدعاء `recognition.start()` أكثر من مرة؟

* **لا يتم استدعاؤها بشكل متكرر أو متوازٍ في نفس الوقت.**
* يتم استدعاؤها **فقط عند ضغط المستخدم يدوياً** على زر الميكروفون عندما تكون الحالة خاملة (`mode !== 'listening'`).
* في كل مرة يُضغط فيها على الزر لبدء التسجيل، يتم إنشاء كائن تسجيل جديد تماماً (`new SR()`) ثم يتم تشغيله بواسطة `recognition.start()` وحفظه في `recRef.current`.
* لا توجد حلقات تكرارية (loops) أو خطافات `useEffect` تقوم باستدعاء `start()` تلقائياً في الخلفية.

---

## 4. هل هناك أي كود يقوم بإيقاف التسجيل تلقائياً؟

* **داخل كود تطبيق React/TypeScript:**
  * **لا يوجد كود إيقاف تلقائي زمني** (مثل `setTimeout` أو مؤقت لإغلاق التسجيل).
  * كود الإيقاف البرمجي الوحيد موجود في دالة المعالجة `process` عند إنهاء التسجيل يدوياً:
    ```typescript
    const process = (text: string) => {
      recRef.current?.stop();
      ...
    };
    ```
    حيث يُستدعى `stop()` عندما يضغط المستخدم على الزر مجدداً لإيقاف التسجيل طواعية.
  * **ملاحظة هامة:** لا يوجد كود تنظيف (cleanup) لإيقاف الميكروفون عند مغادرة الصفحة أو إغلاق النافذة المنبثقة (Modal)، مما يعني أنه إذا أغلق المستخدم النافذة وظل التسجيل قيد العمل، فلن يتم إيقافه برمجياً.

* **من جانب المتصفح (Web Speech API):**
  * متصفحات Chrome و Edge تعتمد على خوادم Google لمعالجة الصوت (خاصة عند اختيار لغات معينة مثل العربية). إذا حدث انقطاع في الاتصال أو لم يتم إرسال أي صوت لفترة قصيرة جداً، يقوم المتصفح تلقائياً بإطلاق حدث `onerror` أو `onend`.
  * بسبب وجود `recognition.onerror = () => setMode('idle')` و `recognition.onend` اللذين يقومان بتغيير الحالة إلى `'idle'` فوراً، فإنه بمجرد أن ينهي المتصفح الجلسة لأي سبب تقني، سينقطع التسجيل في الواجهة فوراً.

## سبب محتمل للمشكلة الحالية (لماذا يقطع فوراً؟)

بالنظر إلى الفروقات بين التعديلين الأخيرين (عبر `git log`):
1. **في التعديل قبل الأخير:** كان الكود يستخدم `continuous = false` و `interimResults = false` وكان يعالج الكلمة بمجرد نطقها ويغلق التسجيل تلقائياً عبر `process(text)`.
2. **في التعديل الأخير:** تم إرجاع الإعدادات إلى `continuous = true` و `interimResults = true` وتم تغيير طريقة قراءة المصفوفة `event.results` باستخدام `Array.from(event.results)`.

**المشكلة الشائعة:** 
في متصفحات Chromium، عند تفعيل `interimResults = true` مع `continuous = true` في بيئة غير مهيأة بالكامل بالـ SSL (أو عند حدوث مشاكل في الشبكة أو في تعامل المتصفح مع حدث `onresult` وتحديث الـ state المتكرر الذي يسبب إعادة رندر سريع جداً للمكون)، قد يطلق المتصفح خطأ صامت مثل `network` أو `no-speech` فيحدث استدعاء لـ `onerror` أو `onend` فوراً، مما يعيد الحالة إلى `idle` وينقطع التسجيل.
